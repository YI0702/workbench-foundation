📄 产品定义 · 一页纸

基座 = 把五个模块串成"一个产品"的形态骨架

不是又一个独立模块。配套可交互 Demo使用。

底稿不是终稿 · 所有定义欢迎讨论共识后迭代 · 维护人：一一

① 定位 · 基座做什么 / 不做什么

✅ 基座做
  • 导航与聚合（首页是入口）
  • 共用能力的载体（5 个模块复用）
  • 用户体验上的统一性
  • 跨模块流转的协议
❌ 基座不做
  • 不是"第六个模块"——不和模块争业务定义
  • 不做模块内的工作流和具体视觉
  • 不是 PM 一个人的看板
  • 不是完成态——会跟模块同步迭代

② PM 的统筹原则 · 最小化干预

PM 只在 1 件事上守底线：入口路径与跳转契约
其余（视觉一致 / 数据口径 / Skills 提报 / 看板卡片 / 关键词路由）都是讨论共识可迭代。
底盘稳，模块才能跑得快。我提的不是约束，是底盘。

③ 基座的双视图 · 形态层 × 能力层

A · 用户视角
基座四层（用户看到的"形态"）
  1. AI 对话入口·语义路由 → 知道要做什么的人
  2. 场景化 Prompt 卡片·6 张高频任务 → 不知道怎么开始的人
  3. 自定义视角看板·赛道/场景/预算/SDK → 业务 owner
  4. Skills 资产晾晒区·核心强定义+分支个人贡献 → 所有人
↓ 调用
B · 技术视角
共用基础设施（5 个模块共享的"能力"）
  • AI 对话路由 · 通用组件库
  • 统一鉴权 · 数据口径共识
  • 操作日志 · Skills 嵌入框架
两者是同一件事的两面——基座是表层，共用基础设施是底层。

④ PM × 模块的协同接口

三种执行模式：🟢模块自定义 / 🟡共建对齐 / 🔴PM兜底。模块内部（工作流、界面、技术）属于 🟢，全权 owner 拍板，本表不涉及。

维度性质谁拍板
AI 路由关键词共建对齐各 owner 提报 → PM 检查冲突 → 共识
首页看板卡片共建对齐owner 设计 → PM 走查 → 上架
Skills 提报上架共建对齐分层审核（见第⑤节）
数据口径共建对齐业务理解共识 → 具体指标 owner 自定义
视觉一致性共建对齐通用组件库 + 设计走查（路演后启动）
入口路径与跳转契约PM 兜底跳转参数协议 PM 定，模块按协议接收
"入口路径与跳转契约"具体指什么

用户从基座跳到模块时 URL 参数和上下文怎么传。例：用户在 AI 对话框输入"金融行业模板效果"→ 跳转洞察归因时 URL 是 ?industry=金融&module=洞察归因，参数命名和协议必须统一，避免 5 个模块用 5 套参数。模块按协议接收实现自由，但协议本身 PM 拍板。

⑤ Skills 管理 · 核心强定义 + 分支共创

导向：工作台对核心使用流程做强定义约束 · Skills 共创满足分支流程的个人贡献。
Skills 是主流程的补充和增强，不是主流程的替代。
分层审核（讨论稿）
类型判断审核路径
A 类涉线上配置 / 跨模块 / 用户隐私PM + 模块 owner + 安全合规
B 类仅影响某模块查询/展示对应模块 owner 审核
C 类个人自用不进资产库，自己空间
⚠️ 关键讨论点：Skill 不只是上架可用——还要和当前模块工作流融合（触发位置、输入输出、失败兜底、版本管理）。本节先给导向，融合细节后续立项讨论。

⑥ 用户主路径 · 4 种使用方式

这 4 条主路径在 Demo 里都能实际点击体验。

01
知道要做什么
AI 对话框输入意图 → 关键词识别 → 路由到模块 → prompt 自动填充
支撑：第 1 层 AI 对话
02
不确定怎么开始
浏览场景卡片（6 张）→ 点击高频任务 → 自动带 prompt 进模块
支撑：第 2 层 场景卡片
03
业务 owner 看大盘
视角切换（赛道/场景/预算/SDK）→ 跨模块聚合数据 → 深入对应模块
支撑：第 3 层 视角看板
04
复用团队资产
浏览 Skills 资产 → 点击触发 → 经关键词路由进模块 → 模块内承载执行
支撑：第 4 层 Skills 共创

⑦ 风险识别 · 语义理解兜底

参考 Anthropic / Cursor / Microsoft Copilot Studio 等业界实践（confidence 70% / fallback rate 阈值 20%）。

失败模式兜底路径
意图无法识别场景卡片兜底·引导用 6 张高频任务卡片
置信度不足（多意图）意图澄清·反问"是 A 还是 B"
重复误解Prompt 模板·"我要 XX → 推荐 YY"
超出范围边界声明·"工作台不处理 XX，建议 YY"
