## 一、为什么做"基座"，不是直接做五个模块

如果把工作台理解成"五个模块的集合"，那就只是一个新的导航站。

我们要做的是**把五个模块用统一的产品语言串起来**——让用户的"工作意图"能流畅地在五个模块之间流转，而不是让用户自己记得"这个数据要去哪个模块查、这个操作要去哪个模块做"。

**基座解决三个问题**：

| # | 问题 | 基座的解 |
|---|------|---------|
| 1 | 用户找不到入口（"这件事去哪个模块？"） | **AI 对话入口**——用自然语言描述意图，自动路由 |
| 2 | 新手不知道从哪开始（"我刚来要先做什么？"） | **场景化 Prompt 卡片**——高频任务一键启动 |
| 3 | 产品 owner 数据散落（"我负责的赛道现在啥情况？"） | **可切换视角的看板**——按赛道/场景/预算/SDK 看全貌 |

## 二、基座的四层结构

```
┌─────────────────────────────────────────────┐
│  第一层：AI 对话入口（语义路由）              │
│  → 自然语言 → 关键词识别 → 分发到模块        │
│  → 服务对象：知道自己要做什么的用户            │
├─────────────────────────────────────────────┤
│  第二层：场景化 Prompt 卡片（高频任务）       │
│  → 6 张高频场景卡片，点击带 prompt 进模块    │
│  → 服务对象：不太确定怎么开始的用户            │
├─────────────────────────────────────────────┤
│  第三层：自定义视角看板（数据汇聚）           │
│  → 赛道/场景/预算/SDK 视角切换               │
│  → 服务对象：负责具体业务的产品 owner         │
├─────────────────────────────────────────────┤
│  第四层：Skills 共创区（封闭定义+开放共创）   │
│  → 官方 Skills + 社区 Skills 双轨           │
│  → 服务对象：所有人——既是使用者也是贡献者     │
└─────────────────────────────────────────────┘
```

**为什么是这四层、为什么按这个顺序？**

- **顶层是 AI 对话**——这是"AI 原生"的产品定调，传递信号"这个工作台和过去不同"
- **第二层是场景卡片**——纯 AI 对话对新手不友好，场景卡片是"扶手"
- **第三层是看板**——首页最重要的是"接下来要干什么"，所以"看历史数据"沉到此处
- **第四层是 Skills**——把团队的"重复造轮子"沉淀为可复用资产，这是落地路线图共识②的核心载体

## 三、关键设计取舍

### 取舍 1：场景卡片以"任务"组织，不以"模块"组织

**对比**：
- 传统做法：首页放"模板生产 / 规则配置 / 洞察归因 / 问题排查 / 全景总览"五个模块图标
- 我们的做法：首页放"新建样式模板 / 配置投放规则 / 洞察样式效果 / 排查异常 case / 查看赛道大盘"五个**任务**

**理由**：用户的认知是"我要做 X 这件事"，不是"我要去 Y 这个模块"。任务化组织让用户跳过"模块认知"直达目的。

### 取舍 2：看板视角支持"自定义"，留 owner 共建气口

**机制**：
- 框架由 PM 定义（视角切换器、卡片栅格、跨视角的口径统一）
- 卡片内容由模块 owner 共建（每个 owner 提报"我希望在首页看板上展示什么"）
- 「+ 自定义视角」按钮预留——当某个模块 owner 觉得需要新维度时，PM 评估后开放

**理由**：避免基座变成"PM 的看板"——它应该是"团队共同的工作视图"。

### 取舍 3：不做后台数据库，先用 mock 数据跑通体验

**当前阶段**：所有数据是 mock，用于验证产品形态
**下一阶段**：接入真实数据底座（H1 横向工作的产物）

**理由**：先把"长什么样"定下来，再讨论"接哪些数据"——形态比数据先行。

### 取舍 4：Skills 采用"封闭定义 + 开放共创"双轨

**机制**：
- **官方 Skills**：必备能力，由产品 + 资深定义，覆盖核心生产/规则/排查环节
- **社区 Skills**：团队成员自由上传，经审计后上架，鼓励"重复造轮子"被沉淀
- **嵌入点**：Skills 不是独立模块，而是嵌入到生产/规则/洞察/排查流程的具体环节里

**理由**：纯封闭定义 = 创新窒息；纯开放 = 质量失控。双轨制平衡稳定与创新。<br>
这也是落地路线图共识②（封闭定义和开放共创要结合）的具体形态。

## 四、基座对五个模块的"接口契约"

每个模块都有三个统一的接口接入基座：

| 接口 | 含义 | 谁负责 |
|------|------|-------|
| **AI 路由关键词** | 模块声明自己响应哪些关键词 | 模块 owner 提报 |
| **看板卡片** | 模块在首页看板上的"代表性卡片"（最多 2 个） | 模块 owner 设计 + PM 审核 |
| **跨模块跳转** | 模块为其他模块提供的"深入查看"链接 | 模块 owner 配合 |

**这三个接口在路演时必须给到**——这是模块定义和基座对齐的最小约定。

## 五、共用基础设施清单

基座除了"长什么样"之外，还要建设**五个模块共用的基础设施**：

| # | 基础设施 | 说明 | 状态 |
|---|---------|------|------|
| 1 | **AI 对话路由** | 语义识别 + 关键词分发 + 上下文继承 | 待立项 |
| 2 | **通用组件库** | 卡片/表格/按钮/图表的统一样式 | 待立项 |
| 3 | **统一鉴权与权限** | 谁能看哪些视角、谁能改哪些规则 | 待立项 |
| 4 | **数据底座** | 跨模块的指标定义、字段对齐、组件特征库 | H1 横向工作 |
| 5 | **操作日志/决策档案** | 谁在什么时间做了什么决定，可追溯 | 项目 Hub 复用 |

> **重要**：这五项不是"PM 一个人建"——而是 PM 提框架，在过程中由各模块共建。

## 六、和五个模块的关系

```
                    [基座首页]
                        │
       ┌────────────────┼────────────────┐
       │   AI对话入口   │   场景卡片入口  │   看板入口
       │       │       │       │        │       │
       ▼       ▼       ▼       ▼        ▼       ▼
  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
  │模板  │ │规则  │ │洞察  │ │问题  │ │全景  │
  │生产  │ │配置  │ │归因  │ │排查  │ │总览  │
  └──────┘ └──────┘ └──────┘ └──────┘ └──────┘
       ↑       ↑       ↑       ↑        ↑
       └───────┴───────┴───────┴────────┘
                    跨模块协同
```

**关键关系**：
- 基座是"前台路由器"——把用户带到正确的模块
- 基座是"全局看板"——汇聚跨模块的关键信息
- 基座是"基础设施"——五个模块复用的能力底盘
- **基座不替代任何模块**——它让模块之间的协作更自然

## 七、对模块 owner 的明确请求

模块 owner 在 5/27 路演前需要给到三件事：

1. **AI 路由关键词清单**（最少 5 个、最多 15 个）
2. **首页看板卡片设计**（最多 2 张，含数据字段、跳转目标）
3. **模块产品定义一页纸 + 可点 Demo**（含对 checklist 关键问题的回应）

> 这三件事是"基座对齐"的最小集——给得到这个，路演就能跑通。

## 九、未尽事项 / 待讨论

- AI 对话路由的具体技术实现（LLM 直接做 vs 关键词规则 + LLM 兜底）
- 看板视角的"自定义"开放程度（开放到 owner 还是开放到所有用户）
- 跨模块跳转的"上下文携带"机制（怎么把当前查询条件带到下个模块）
- 基础设施的"建设节奏"（一次性建完 vs 跟着模块需求按需建）

> 这些都不在周一会议讨论范围——周一只对齐"形态"，技术细节留到研发 KO。
