## 一、定位与边界

### 基座是什么

**基座是五个模块共用的"产品形态层"**——它定义了用户进入工作台的第一眼看到什么、怎么开始工作、跨模块时如何流转。

**一句话定义**：
> 基座是把五个模块串成"一个产品"的形态骨架，不是又一个独立模块。

### 基座不是什么（重要 · 划清边界）

- ❌ 不是"第六个模块"——不和五个模块争具体业务定义
- ❌ 不是"功能集合"——基座只做导航、聚合、共用能力的载体
- ❌ 不是"PM 一个人的看板"——基座的多数定义需要和模块 owner 共识
- ❌ 不是"完成态"——它会跟着模块定义同步迭代

### PM（一一）的统筹原则 · 最小化干预

> **PM 只在两件事上守底线：视觉一致 + 入口路径。其余都是讨论共识可迭代的。**

**为什么这样**：
- 守"视觉一致"——五个模块拼起来是一个产品，不是五个小作坊
- 守"入口路径"——用户跳转的语义和参数契约必须统一，不然会乱
- 其余放给 owner——我的工作量降到最低，模块灵活度最大化
- 我提的不是约束，是底盘——**底盘稳，模块才能跑得快**

## 二、双视图 · 基座（四层）× 共用基础设施

基座有两个视图——**用户视角**和**技术视角**——它们看的是同一件事的两面。

### 视图 A · 基座四层（用户视角）

```
第一层 · AI 对话入口（语义路由）
   └─ 用户用自然语言描述意图，系统识别后跳转对应模块
   └─ 服务对象：知道自己要做什么的用户

第二层 · 场景化 Prompt 卡片（高频任务）
   └─ 6 张高频场景卡片，点击带 prompt 进对应模块
   └─ 服务对象：不太确定怎么开始的用户

第三层 · 自定义视角看板（数据汇聚）
   └─ 赛道 / 场景 / 预算 / SDK 视角切换
   └─ 服务对象：负责具体业务的产品 owner

第四层 · Skills 资产晾晒区（封闭定义+开放共创）
   └─ 资产维护与展示，触发链路 by 关键词路由到对应模块
   └─ 服务对象：所有人——既是使用者也是贡献者
```

### 视图 B · 共用基础设施清单（技术视角）

| # | 基础设施 | 说明 | 谁主导 |
|---|---------|------|-------|
| 1 | **AI 对话路由** | 语义识别 + 关键词分发 + 上下文继承 | PM 提框架 + 各模块提报关键词 |
| 2 | **通用组件库** | 卡片/表格/按钮/图表的最小一致集 | PM 守底线（最小约束） |
| 3 | **统一鉴权与权限** | 谁能看哪些视角、谁能改哪些规则 | 共识定义，PM 协调 |
| 4 | **数据口径共识** | 跨模块的业务理解 + 可用性评估导向 | 共识导向，模块自定义具体指标 |
| 5 | **操作日志/决策档案** | 谁在什么时间做了什么决定 | 复用 Nexus 的决策档案能力 |
| 6 | **Skills 嵌入框架** | Skills 的注册/发现/调度协议 | PM 提框架 + 各模块负责自己 Skill 的工作流 |

### 两者关系（关键）

```
┌─ 基座四层（用户看到的「形态」）─────────────┐
│  AI 对话 / 场景卡片 / 看板 / Skills 晾晒    │
│      ↑ 调用                                 │
├─ 共用基础设施（五个模块共享的「能力」）────┤
│  AI 路由 / 组件库 / 鉴权 / 数据底座 /        │
│  操作日志 / Skills 框架                      │
└─────────────────────────────────────────────┘
```

**基座是表层 · 共用基础设施是底层**——基座的某些功能调用共用基础设施提供的能力。**两者一起服务于"五个模块拼成一个产品"这件事**。

## 三、PM 与各模块的协同接口（核心）

下面六个维度，是 PM 和模块 owner 要对齐的所有事。**绿色 = 讨论共识可迭代 / 红色 = PM 守底线**。

| 协同维度 | PM 给什么 | 模块回应什么 | 性质 |
|---------|---------|------------|------|
| **AI 路由关键词** | 关键词池注册机制 + 语义路由 | 本模块响应的关键词列表（5-15 个） | 🟢 讨论共识 |
| **首页看板卡片** | 卡片栅格 + 渲染规范 + 视角切换 | 1-2 张本模块代表性卡片设计 | 🟢 讨论共识 |
| **Skills 提报** | 资产晾晒区 + 嵌入协议 | 本模块可贡献的 Skill 列表 + 各 Skill 的底层工作流和界面 | 🟢 讨论共识 |
| **数据口径** | 跨模块业务理解 + 可用性评估导向 | 本模块上报的具体指标定义 | 🟢 强讨论 |
| **视觉一致性** | 通用组件库（最小约束集） | 模块内自由发挥 | 🔴 PM 守底线 |
| **入口路径与跳转契约** | 基座→模块的跳转语义 + 参数契约 | 模块入口参数接收实现 | 🔴 PM 守底线 |

### Skills 机制的特别说明

> **首页只做资产晾晒和发现 · 触发链路 by 关键词路由到对应模块 · 底层工作流和界面由对应模块负责**

- **首页**：展示官方/社区 Skills 列表，提供搜索和分类
- **触发**：用户在首页点击 Skill → 经关键词路由跳转到对应模块 → 模块内承载该 Skill 的具体执行界面
- **责任划分**：模块 owner 同时负责本模块的底层工作流建设 + 模块界面建设（不论从首页还是模块入口触发，都进同一套工作流）

**这样设计的好处**：基座不需要知道每个 Skill 怎么跑——只需要知道它属于哪个模块、入口长啥样。

## 四、对模块 owner 的明确请求（5/27 周二前提报）

### 必交付（三个）

| # | 提报项 | 描述 | 提报粒度 |
|---|-------|------|---------|
| 1 | **AI 路由关键词清单** | 本模块响应哪些自然语言意图 | 5-15 个关键词，每个附触发示例 |
| 2 | **首页看板卡片设计** | 你希望在首页放 1-2 张什么卡片 | 字段、数据来源、跳转目标 |
| 3 | **Skills 提报清单** | 本模块可贡献的 Skill 列表 | 至少 1 个，含 Skill 名、用途、触发关键词、是否官方 |

### 建议交付（三个）

| # | 建议项 | 描述 | 何时给 |
|---|-------|------|-------|
| 4 | **数据口径需求** | 本模块希望基座共识哪些数据导向 | 5/29 路演前 |
| 5 | **入口路径需求** | 用户从基座跳到本模块时希望带什么参数 | 5/29 路演前 |

## 五、用户使用主路径（基座视角）

> 这部分回应 Review 清单中"业务主流程怎么在 Demo 里反映"的关键问题。

### 主路径 1 · 知道自己要做什么的用户

```
打开工作台
   ↓
首页 AI 对话框输入意图（自然语言）
   ↓
基座识别关键词 → 路由到对应模块
   ↓
进入模块界面，prompt 已自动填充
   ↓
开始具体工作
```

**支撑**：基座第一层（AI 对话入口） · 共用基础设施①（AI 对话路由）

### 主路径 2 · 不太确定怎么开始的用户

```
打开工作台
   ↓
首页浏览场景卡片（6 个高频任务）
   ↓
点击对应任务卡片 → 自动带 prompt 进入模块
   ↓
模块内承接 prompt，进入工作流
```

**支撑**：基座第二层（场景化 Prompt 卡片）

### 主路径 3 · 负责具体业务的产品 owner

```
打开工作台
   ↓
首页看板按视角切换（赛道/场景/预算/SDK）
   ↓
查看跨模块聚合的关键数据
   ↓
点击具体卡片的"查看完整 →" 深入对应模块
```

**支撑**：基座第三层（自定义视角看板） · 共用基础设施④（数据口径共识）

### 主路径 4 · 想复用团队资产的用户

```
打开工作台
   ↓
浏览 Skills 资产晾晒区（官方 + 社区）
   ↓
找到合适的 Skill，点击触发
   ↓
基座经关键词路由到对应模块
   ↓
模块内承载 Skill 的具体执行界面
```

**支撑**：基座第四层（Skills 资产晾晒） · 共用基础设施⑥（Skills 嵌入框架）

## 六、定义边界回应（Review 清单自查）

> 主动回应 Review 清单中的关键检查项——示范一份"产品定义如何回应 Review"。

### 模块边界

- **基座做什么**：导航、聚合、共用能力的载体；用户体验的统一性
- **基座不做什么**：不做任何模块的具体业务逻辑；不做模块内的工作流；不做模块独有的视觉
- **和相邻模块的关系**：基座是模块的"上游入口"，不是模块的"上级"

### 历史逻辑兼容

- **必迁**：原平台首页的核心入口、跨模块跳转链路、用户角色权限
- **缓迁**：原首页的固定看板（改造为可切换视角）
- **不迁**：原首页的若干静态展示组件（功能不再需要）

### AI 原生性体现

- 第一层 AI 对话入口本身就是 AI 原生的——传统工作台不会用对话框做主入口
- 看板的"迭代建议"卡片由 AI 生成
- Skills 区的语义路由由 AI 完成
- **AI 在基座是"主角"还是"辅助"**：主辅混合——AI 对话和 Skills 路由是主角，看板和场景卡片是辅助

### 风险识别

- **最大风险**：AI 路由命中率不达预期，用户回到"找模块"的旧体验——应对：场景卡片做兜底
- **次大风险**：模块 owner 提报的看板卡片视觉不一致——应对：通用组件库 + PM 走查
- **不可控环节**：Skills 上架审计机制还在共识中
