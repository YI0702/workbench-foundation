## 一、定位与边界

### 基座是什么

**基座是五个模块共用的"产品形态层"**——它定义了用户进入工作台的第一眼看到什么、怎么开始工作、跨模块时如何流转。

**一句话定义**：
> 基座是把五个模块串成"一个产品"的形态骨架，不是又一个独立模块。

### 基座不是什么（重要 · 划清边界）

- ❌ 不是"第六个模块"——不和五个模块争具体业务定义
- ❌ 不是"功能集合"——基座只做导航、聚合、共用能力的载体
- ❌ 不是"PM 一个人的看板"——基座的多数定义需要和模块 owner 共识
- ❌ 不是"完成态"——它会跟着模块定义同步迭代

### PM（一一）的统筹原则 · 最小化干预

> **PM 只在一件事上守底线：入口路径与跳转契约。其余都是讨论共识可迭代的。**

**为什么这样**：
- 守"入口路径"——用户跳转的语义和参数协议必须统一，不然各模块拼不起来
- 视觉一致性 / 数据口径 / Skills 提报 / 看板卡片 / 关键词路由——这些都是讨论共识可迭代的
- 我提的不是约束，是底盘——**底盘稳，模块才能跑得快**
- 我的工作量降到最低——避免成为团队节奏的瓶颈

## 二、双视图 · 基座（四层）× 共用基础设施

基座有两个视图——**用户视角**和**技术视角**——它们看的是同一件事的两面。

### 视图 A · 基座四层（用户视角）

```
第一层 · AI 对话入口（语义路由）
   └─ 用户用自然语言描述意图，系统识别后跳转对应模块
   └─ 服务对象：知道自己要做什么的用户

第二层 · 场景化 Prompt 卡片（高频任务）
   └─ 6 张高频场景卡片，点击带 prompt 进对应模块
   └─ 服务对象：不太确定怎么开始的用户

第三层 · 自定义视角看板（数据汇聚）
   └─ 赛道 / 场景 / 预算 / SDK 视角切换
   └─ 服务对象：负责具体业务的产品 owner

第四层 · Skills 资产晾晒区（核心强定义 + 分支个人贡献）
   └─ 核心使用流程由工作台强定义约束
   └─ 分支流程通过 Skills 形式开放个人贡献
   └─ 服务对象：所有人——既是使用者也是贡献者
```

### 视图 B · 共用基础设施清单（技术视角）

| # | 基础设施 | 说明 | 谁主导 |
|---|---------|------|-------|
| 1 | **AI 对话路由** | 语义识别 + 关键词分发 + 上下文继承 | PM 提框架 + 各模块提报关键词 |
| 2 | **通用组件库** | 卡片/表格/按钮/图表的最小一致集 | PM + 模块 owner + 设计共建 |
| 3 | **统一鉴权与权限** | 谁能看哪些视角、谁能改哪些规则 | 共识定义，PM 协调 |
| 4 | **数据口径共识** | 跨模块的业务理解 + 可用性评估导向 | 共识导向，模块自定义具体指标 |
| 5 | **操作日志/决策档案** | 谁在什么时间做了什么决定 | 复用 Nexus 的决策档案能力 |
| 6 | **Skills 嵌入框架** | Skills 的注册/发现/调度协议 | PM 提框架 + 各模块负责自己 Skill 的工作流 |

### 两者关系（关键）

```
┌─ 基座四层（用户看到的「形态」）─────────────┐
│  AI 对话 / 场景卡片 / 看板 / Skills 晾晒    │
│      ↑ 调用                                 │
├─ 共用基础设施（五个模块共享的「能力」）────┤
│  AI 路由 / 组件库 / 鉴权 / 数据底座 /        │
│  操作日志 / Skills 框架                      │
└─────────────────────────────────────────────┘
```

**基座是表层 · 共用基础设施是底层**——基座的某些功能调用共用基础设施提供的能力。**两者一起服务于"五个模块拼成一个产品"这件事**。

## 三、PM 与各模块的协同接口（核心）

### 三种执行模式

| 性质 | 执行模式 | 决策方式 | 谁拍板 |
|------|---------|---------|-------|
| 🟢 **模块自定义** | owner 直接定义，PM 不介入 | owner 拍板 | 模块 owner |
| 🟡 **共建对齐** | owner 提报 + PM 协调 + 跨模块对齐 | 讨论后共识 | PM 主持，多方签字 |
| 🔴 **PM 兜底** | PM 守住底线规则，模块在规则内自由 | PM 拍板 | PM |

> **重要说明**：模块内部的事（工作流、界面、技术选型）属于 🟢 模块自定义，owner 全权拍板，本节不涉及。本节只列**跨模块协同接口**。

### 6 个协同维度

| # | 协同维度 | 性质 | 执行说明 |
|---|---------|------|---------|
| 1 | **AI 路由关键词** | 🟡 共建对齐 | 各 owner 提报 → PM 检查冲突 → 共识终版 |
| 2 | **首页看板卡片** | 🟡 共建对齐 | 各 owner 设计 → PM 走查 → 视觉规范对齐 → 上架 |
| 3 | **Skills 提报与上架** | 🟡 共建对齐 | 见第四节专门说明 |
| 4 | **数据口径** | 🟡 共建对齐 | 业务理解层共识 → 具体指标 owner 自定义 |
| 5 | **视觉一致性** | 🟡 共建对齐 | 通用组件库（最小集）+ 设计协同走查 |
| 6 | **入口路径与跳转契约** | 🔴 PM 兜底 | 跳转参数协议 PM 定 |

### 为什么入口路径要 PM 兜底

> "入口路径与跳转契约" 听起来抽象，具体指——

**用户从基座跳到模块时，URL 参数和上下文怎么传**：
- 例：用户在 AI 对话框输入"金融行业模板效果"，跳转到洞察归因模块时，URL 应该是 `?industry=金融&module=洞察归因`，不是各模块各自定义一套参数
- 这套**参数协议**必须统一——否则用户在 A 模块的查询条件传到 B 模块就丢了
- 模块按协议接收实现自由，但**协议**本身 PM 拍板

如果这个不守底线——五个模块跨模块跳转就会变成五套不同的参数命名、五种丢失上下文的方式。

## 四、Skills 管理流程

### 整体导向

> **工作台对核心使用流程和规范做强定义约束 · Skills 共创满足分支流程的个人贡献**

- **核心使用流程**：模板生产→规则配置→洞察归因→问题排查的端到端主流程，由工作台强定义、不开放共创
- **Skills 共创**：分支流程的个人贡献，比如"批量导出某类规则到 Excel"、"快速对比两个模板的 CTR"等局部高频但不主流程的能力
- **Skills 共创的定位**：**主流程的补充和增强，不是主流程的替代**

### 上架流程方案 · 方案 B 分层审核（讨论稿）

```
个人开发 Skill → 自评分级（A/B/C）→ 对应审核路径 → 上架到资产库
```

| 类型 | 判断标准 | 审核路径 |
|------|---------|---------|
| **A 类**（高风险/全局影响） | 涉及线上配置变更、跨模块、用户隐私 | PM + 对应模块 owner + 安全/合规审核 |
| **B 类**（模块级、不涉规则变更） | 仅影响某一模块的查询/展示 | 对应模块 owner 审核即可 |
| **C 类**（个人自用工具） | 不进入公共资产库 | 自己空间，不审核 |

### 审核之外 · Skills 与现有流程的融合（关键讨论点）

> **个人贡献的 Skill 不只是"上架可用"——它需要能和当前的模块工作流做融合**：

- **Skill 的触发位置**：嵌入到模块工作流的哪一步？
- **Skill 的输入输出**：和模块的数据格式如何对齐？
- **Skill 的失败兜底**：如果 Skill 出错，工作流回退到哪里？
- **Skill 的版本管理**：模块工作流升级后，旧 Skill 怎么办？

这些融合细节**先给导向、不穷举细节**——具体方案 PM + 资深 + 各模块 owner 周一会议后单独立项讨论。

### 现阶段先要明确的（5/29 路演前）

- 是否同意采用方案 B 分层审核
- A/B/C 三类的判断标准是否清晰
- "Skills 是分支流程补充、不是主流程替代"的定位是否团队共识

## 五、对模块 owner 的明确请求（5/29 路演前提报）

### 6 项交付清单

| # | 交付项 | 重要性 | 描述 | 提报粒度 |
|---|-------|------|------|---------|
| 1 | **AI 路由关键词清单** | 必要 | 本模块响应哪些自然语言意图 | 5-15 个关键词，每个附触发示例 |
| 2 | **首页看板卡片设计** | 必要 | 你希望在首页放 1-2 张什么卡片 | 字段、数据来源、跳转目标 |
| 3 | **Skills 提报清单** | 必要 | 本模块可贡献的 Skill 列表 | 至少 1 个，含 Skill 名、用途、触发关键词、A/B/C 类型 |
| 4 | **数据口径需求** | 建议 | 本模块希望基座共识哪些数据导向 | 业务理解层的需求点 |
| 5 | **入口路径需求** | 建议 | 用户从基座跳到本模块时希望带什么参数 | 参数列表 + 示例 |
| 6 | **历史能力承接说明** | 建议 | 你这个模块对应的能力原来在哪个旧平台、新模块如何承接 | 一段话说清楚 |

> 这 6 项 5/29 路演前一次性给到。**不是要求 owner 立刻完成所有，是路演当天展示这些就足够**。

## 六、用户使用主路径（基座视角）

> 这部分回应 Review 清单中"业务主流程怎么在 Demo 里反映"的关键问题。

### 主路径 1 · 知道自己要做什么的用户

```
打开工作台 → 首页 AI 对话框输入意图（自然语言）
           → 基座识别关键词 → 路由到对应模块
           → 进入模块界面，prompt 已自动填充
           → 开始具体工作
```
**支撑**：基座第一层（AI 对话入口） · 共用基础设施①（AI 对话路由）

### 主路径 2 · 不太确定怎么开始的用户

```
打开工作台 → 首页浏览场景卡片（6 个高频任务）
           → 点击对应任务卡片 → 自动带 prompt 进入模块
           → 模块内承接 prompt，进入工作流
```
**支撑**：基座第二层（场景化 Prompt 卡片）

### 主路径 3 · 负责具体业务的产品 owner

```
打开工作台 → 首页看板按视角切换（赛道/场景/预算/SDK）
           → 查看跨模块聚合的关键数据
           → 点击具体卡片"查看完整 →" 深入对应模块
```
**支撑**：基座第三层（自定义视角看板） · 共用基础设施④（数据口径共识）

### 主路径 4 · 想复用团队资产的用户

```
打开工作台 → 浏览 Skills 资产晾晒区（官方 + 社区）
           → 找到合适的 Skill，点击触发
           → 基座经关键词路由到对应模块
           → 模块内承载 Skill 的具体执行界面
```
**支撑**：基座第四层（Skills 资产晾晒） · 共用基础设施⑥（Skills 嵌入框架）

## 七、跨职能协同 · 设计 / 研发

### 与设计的协同 · 精简合作 + 最快跑通流程

**导向**：设计**只做高价值环节**——基座视觉规范 + 通用组件库 + 路演反馈走查 + 上线前 polish。模块内的具体视觉由 owner 用 AI 工具直出。

**具体分工**：

| 设计做什么 | 设计不做什么 |
|----------|------------|
| ✅ 通用组件库的最小约束集（卡片/表格/按钮/图表） | ❌ 跟着模块定义阶段做细节稿 |
| ✅ 基座首页视觉规范（颜色、字体、间距） | ❌ 替模块 owner 出 Demo 视觉 |
| ✅ 5/29 路演当天做视觉一致性走查 + 反馈 | ❌ block 模块定义节奏 |
| ✅ 上线前关键模块的视觉 polish | ❌ 全程跟单个模块 |

**这样设计的好处**：
- 设计专注高价值环节，不被节奏拖累
- 模块 owner 用 AI 工具直出 Demo，速度优先于细节
- 设计在路演时统一走查，能快速发现"五个模块拼起来不一致"的地方

**待和设计沟通的点**：通用组件库的最小约束集具体长什么样、视觉规范的颗粒度（待和设计 1on1）。

### 与研发的协同 · 三件套交付契约

**6/01 研发 KO 时给到研发的三件套**：
- ✅ 可交互产品 Demo（HTML/前端代码）
- ✅ 一页纸业务逻辑定义（含 Review 清单关键问题回答）
- ✅ 决策档案（沉淀在 Nexus）

**关键信号**：
- 研发不需要等 PRD 才能开工——三件套就是新交付契约
- 研发可视基座 Demo 中的前端代码作为参考骨架
- 研发反馈通过 Nexus 的"关键反馈与协同"标准化记录

## 八、风险识别 · 语义理解兜底机制

> 参考 Anthropic、Cursor、Microsoft Copilot Studio 等业界实践

### 4 类失败模式 + 4 种兜底路径

| # | 失败模式 | 兜底路径 |
|---|---------|---------|
| 1 | **意图无法识别**（关键词都没匹配上） | **场景卡片兜底** · 引导用户用 6 张高频任务卡片 |
| 2 | **置信度不足**（多个意图都有可能） | **意图澄清** · 反问"你说的是 A 还是 B？"做二选一确认 |
| 3 | **重复误解**（用户改写后仍然不准） | **Prompt 模板引导** · 给用户提供"我要 XX → 推荐输入 YY"的模板 |
| 4 | **超出范围**（用户问的事不在工作台范围内） | **明确边界声明** · 告诉用户"工作台不处理 XX，建议去 YY" |

### 主要风险与应对

| 风险 | 等级 | 应对 |
|------|------|------|
| 语义理解不及预期 | 🔴 高 | 4 种兜底路径全部预置；监控 fallback rate（业界推荐阈值：低于 20%） |
| 模块拼接视觉不一致 | 🟡 中 | PM 或资深做端到端走查；通用组件库的最小约束 |
| Skills 与现有流程融合复杂 | 🟡 中 | 先做 A/B/C 分层审核+导向；具体融合机制后续立项讨论 |
| 跨模块跳转上下文丢失 | 🟡 中 | 入口路径协议 PM 兜底（见第三节） |

## 九、权限分层与协同（讨论稿）

### 信息透明 · 可写权限分层

> **导向**：信息全部公开透明（任何人都能看），可写权限按角色分层。

| 角色 | 可写范围 | 可读范围 |
|------|---------|---------|
| **流量商业产品同学** | 全部功能可写可配 | 全部 |
| **预算商业产品同学** | 预算视角出发的局部改造 ⚠️ | 全部 |
| **其他产运** | 看数据 + 查进度，写权限较低 | 全部 |

### 预算商业产品同学的写权限限制（讨论点）

> ⚠️ 待讨论：**预算商业产品同学的可写范围会受流量大盘影响**——预算视角的改造如果会**影响流量大盘的全局配置**，则不允许直接修改，需要走流量商业产品同学审批。

具体限制规则待 5/29 路演反馈后细化。

### 全员可读的具体内容

包括但不限于：
